> 用途：将本文件连同项目目录提供给新的 AI 会话，使其无需回看原始对话即可继续设计、开发、排错和测试。
>
> 最后更新：2026-07-15

## 1. 项目定位

PointPad 是一个面向桌游场景的多人、分轮、累加型计分网页。

这是一个纯 Vibe Coding 项目，产品构思、界面设计和代码实现均通过人与 AI 的持续对话协作完成。

项目的明确边界：

- 仅支持累加型计分，不实现其他计分规则。
- 每轮分数可以为正数、零或负数。
- 不需要计时器。
- 不需要登录、后端、数据库或云端同步。
- 不使用框架、构建工具或第三方前端依赖。
- 只使用 HTML、CSS 和原生 JavaScript。
- 必须可以作为纯静态网站托管在 GitHub Pages。
- 使用相对资源路径，部署后不应依赖固定域名。
- 主要使用环境是手机，同时兼容平板和桌面浏览器。

## 3. 技术结构

页面按照以下顺序加载普通脚本，顺序不可随意调整：

1. `js/config.js` 创建 `window.SCOREKEEPER_CONFIG`。
2. `js/storage.js` 创建 `window.ScoreStore`。
3. `js/scoring.js` 创建 `window.ScoreMath`。
4. `js/app.js` 读取上述对象并启动应用。

所有脚本都使用 IIFE 隔离作用域，没有 ES Module，因此既可以通过静态服务器运行，也可以直接打开 `index.html`。

应用是单页状态机，包含三个主视图：

- `setup`：创建新对局或恢复已有对局。
- `game`：录入分数、查看累计排名和操作历史。
- `results`：展示最终成绩。

辅助界面使用原生 `<dialog>`：

- 历史记录对话框。
- 对局设置对话框。
- 通用确认对话框。

## 5. 数据模型

核心 `state` 结构如下：

```js
{
  version: 1,
  screen: "setup" | "game" | "results",
  gameName: string,
  players: [
    {
      id: string,
      name: string,
      color: "#rrggbb",
      active: boolean
    }
  ],
  rounds: [
    {
      id: string,
      scores: {
        [playerId]: number
      },
      createdAt: ISODateString
    }
  ],
  draft: {
    [playerId]: number
  },
  allowDecimals: boolean,
  decimalPlaces: number,
  ended: boolean,
  undoStack: [
    {
      label: string,
      data: StateSnapshot
    }
  ],
  createdAt: ISODateString | null,
  updatedAt: ISODateString | null
}
```

数据原则：

- `rounds` 是已提交的事实数据。
- `draft` 只保存当前尚未提交的一轮。
- 总分不单独保存，而是根据 `rounds` 动态计算，避免数据不一致。
- 玩家顺序由 `state.players` 数组顺序决定。
- 排名是派生数据，不改变 `state.players`。
- 保存的数据会经过 `normalizeState()` 校验和补全。

## 6. 分数与排名规则

### 6.1 分轮提交

每位玩家先录入本轮分数，点击“提交第 N 轮”后才写入 `rounds`。

如果所有参与玩家本轮都是 0 分，会弹出确认，但仍允许提交。

暂停参与的玩家提交本轮时记为 0 分。

### 6.2 累计分数

累计分数只包含已经提交的轮次。

这是重要的最终需求：

- 本轮草稿不得加入累计排行榜。
- 本轮草稿不得改变计分卡中的“分数”。
- 只有提交当前轮后，排行榜和计分卡累计分数才更新。

### 6.3 排名

排行榜标题为“已累计分数排名”。

排名按累计分数从高到低排列，采用竞赛排名：

```text
50 分 → 第 1 名
50 分 → 第 1 名
42 分 → 第 3 名
```

排行榜会排序，但玩家计分卡绝不能跟随排行榜重新排序。计分卡始终按照设置中的玩家顺序显示。

### 6.4 小数

- 新对局是否允许小数由配置决定。
- 对局中可以在设置页最下方开启或关闭小数。
- 小数位数可以在前端调整，范围由配置的 `minDecimalPlaces` 和 `maxDecimalPlaces` 控制。
- 小数位数随对局保存。
- 关闭小数时，已有小数分数会四舍五入为整数。
- 减少小数位数时，草稿和历史数据会按新精度归一化。
- `ScoreMath.normalizeScore()` 用于降低浮点累加误差，例如避免显示 `0.30000000000000004`。

## 7. 玩家管理

初始创建页面支持：

- 2～12 位玩家。
- 编辑姓名。
- 编辑颜色。
- 添加或删除玩家。
- 姓名不得重复；空姓名会转换为默认名称。

对局进行中的设置页支持：

- 修改对局名称。
- 修改玩家姓名和颜色。
- 上移、下移玩家，调整计分卡录入顺序。
- 通过“参与”复选框暂停或恢复玩家。
- 添加新玩家；此前轮次自动补 0 分。
- 永久删除玩家。

删除规则：

- 永久删除已有玩家前需要确认。
- 保存设置后，会删除该玩家的草稿和所有历史轮次分数。
- 至少保留 `minPlayers` 位玩家。
- 整次设置修改会先写入撤销快照，因此保存后仍可在计分页撤销恢复。
- 暂停玩家与永久删除不同：暂停会保留历史成绩。

当前移动端设置布局要求：删除按钮位于“参与”复选框左侧，两者作为同一组显示。

## 8. 撤销与历史记录

最多保留 `maxUndoSteps` 个撤销快照，当前为 30。

可撤销的主要操作包括：

- 快捷加减分。
- 手动输入本轮分数。
- 清空本轮。
- 提交轮次。
- 修改或删除历史轮次。
- 修改玩家设置、顺序、小数设置和永久删除玩家。
- 结束对局或返回对局。

历史记录页面：

- 以表格显示每轮每人的分数。
- 可以修改任意历史分数。
- 可以删除整轮。
- 修改后自动重新计算累计总分和排名。
- 删除中间轮次后，界面按数组顺序自动重新编号。

## 9. 本地保存与恢复

- 使用 `localStorage`，封装在 `js/storage.js`。
- 对局开始后每次关键操作自动保存。
- 本轮尚未提交的草稿也会保存。
- 页面刷新或关闭后可以继续。
- 创建页面会显示“发现未完成的对局”，允许继续或放弃。
- 顶部原先存在绿色圆点和“本地自动保存”文字，最终需求已将这两者完全移除。
- 自动保存本身仍然存在，只是不显示状态。
- 清理浏览器站点数据会删除保存的对局。

开发时如果新默认配置看起来没有生效，优先检查是否恢复了旧 localStorage 数据。可以在 Chrome 开发者工具的 Application → Local Storage 中删除对应键。

## 10. 页面和交互现状

### 10.1 创建对局页

- 左侧或上方展示海盗船图片。
- 右侧或下方展示创建对局表单。
- 默认四名玩家。
- 如果有未结束对局，会显示恢复提示。

### 10.2 计分页

顶部：

- 对局名称。
- “当前第 N 轮”标签。
- 历史记录按钮。
- 设置按钮。
- “结束对局”按钮。

“结束对局”要求：

- 桌面和移动端都显示完整文字，不使用图标替代。
- 使用与“提交本轮”一致的主色按钮风格。

累计排名：

- 标题为“已累计分数排名”。
- 显示排名、玩家名称和累计分数。
- 不包含本轮草稿。

计分卡：

- 显示玩家名称、累计名次、累计“分数”和“本轮分数”。
- “分数”不随本轮录入实时变化。
- 快捷按钮默认为 `-10、-5、-1、+1、+5、+10`。
- 可以直接输入数字。

底部计分栏：

- 桌面端为固定底栏。
- 移动端必须单行显示，顺序从左到右严格为：
  1. 本轮合计分数。
  2. 撤销。
  3. 清空本轮。
  4. 提交。
- 按钮文字不得换行。

### 10.3 结果页

- 显示冠军或并列冠军。
- 显示前三名卡片和最高单轮分数。
- 显示完整成绩表。

最终成绩表的顺序是明确需求：

1. 第一行显示玩家名称。
2. 第二行直接显示每位玩家的总分。
3. 后续再依次显示第 1 轮、第 2 轮等每轮得分。
4. 不再在表格最底部重复显示总分。

结果页操作：

- 返回修改。
- 相同玩家再来一局。
- 创建全新对局。

相同玩家再来一局会保留小数开关和小数位数，并重新激活玩家。

## 11. 首页海盗船素材

当前直接使用 `asset/ship.png`：

- 尺寸：1536×1024。
- 文件约 2.8 MB。
- PNG 实际具有透明背景。
- 船体线条是接近黑色的深色线条。
- CSS 使用 `filter: invert(1)` 将线条反转为浅色，以适应深色页面。
- 图片周围没有边框、虚线或独立卡片背景。
- 图片直接悬浮在网页整体背景上。
- 图片容器的实际布局宽度不超过内容列；为抵消原图四周透明留白，图片在 `transform` 动画中视觉放大。
- 动画只作用于整张图片的 `transform`，包含轻微移动、缩放和旋转，不使用超过 100% 的容器宽度。
- 不使用 Canvas、视频、滤镜动画或 JavaScript 定时器。
- `prefers-reduced-motion: reduce` 下停止动画。

不要恢复之前的内联 SVG 海盗船，除非用户明确要求。

## 12. 响应式布局要求

主要断点：

- 900px：创建页从双栏变成单栏；工具栏重排。
- 720px：计分卡变成单列；底栏进入移动布局；轮次标签居中。
- 520px：进一步压缩工具栏和设置行。
- 380px：为极窄屏幕降低间距和字号。

必须保持：

- 整个页面不能出现需要左右拖动的横向滚动。
- 宽成绩表可以在自己的 `.table-wrap` 内横向滚动。
- 海盗船为了补偿透明留白可以视觉上超出内容列，但不得扩大页面滚动宽度。
- 移动端“当前第 N 轮”标签独占工具栏第二行并水平居中。
- 移动端“结束对局”显示完整文字。
- 移动端底栏四项保持单行且不换行。
- 移动端设置页的删除按钮紧邻“参与”复选框左侧。

全局已经使用 `overflow-x: hidden` 和 `overflow-x: clip` 作为保护，但继续开发时仍应修复实际溢出的组件，不要只依赖裁切掩盖问题。

已修复过的移动端溢出问题：

- `html` 和 `body` 必须保持 `width: 100%` 与 `max-width: 100%`，不得为标题栏按钮人为减去固定宽度，否则会在右侧留下长条空白。
- `.setup-layout` 和 `.intro-panel` 已设置 `min-width: 0`，允许 Grid 内容正常收缩。
- `.pirate-scene` 使用 `width: 100%` 和有限 `max-width`，移动端不得恢复 `135%` 或 `145%` 宽度。

## 18. 后续开发原则

- 修改前先阅读 `README.md`、`AI_HANDOFF.md` 和相关源文件。
- 维持零依赖和 GitHub Pages 兼容性，除非用户明确改变技术方向。
- 站点基础参数继续集中在 `js/config.js`，不要散落硬编码配置。
- 相对独立功能继续拆分到对应 JS/CSS 文件，不要把所有逻辑塞回 `index.html`。
- 保持用户已有 localStorage 数据兼容；修改数据结构时增加版本处理或更换存储键。
- 玩家录入顺序与排行榜顺序必须分离。
- 累计排名和累计分数不得包含未提交草稿。
- 破坏性操作必须确认，并尽量纳入撤销。
- 移动端优先，但不要破坏桌面端。
- 不要恢复计时器、复杂计分模式、顶部保存绿点或移动端结束图标。
- 新增动画时优先使用少量 `transform`/`opacity`，并支持 `prefers-reduced-motion`。
- 每次修改后至少进行 JS 语法、HTML 引用和 CSS 结构检查。
